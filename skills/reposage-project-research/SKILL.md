# Reposage 项目调研

使用这个技能围绕一个开源仓库做分层研究。默认先用公开文档和仓库页面回答问题；只有用户明确要求记录、生成文档、源码架构或完整档案时，才写入 `docs/projects/` 或浅克隆源码。

## 触发场景

当用户给出仓库 URL 或 `owner/repo`，并要求项目调研时使用，例如：

- `研究一下 https://github.com/example/project`
- `帮我看看这个开源项目是干什么的`
- `分析一下 owner/repo`
- `这个项目值得用吗`
- `这个项目怎么安装`
- `给这个项目做一份简报`
- `深入看看这个项目的源码结构`

不要用这个技能处理 GitHub Trending 汇总请求；趋势简报使用 `reposage-github-trending-brief`。

如果用户只贴仓库地址，没有给出具体问题或期望产物，先简短询问用户想要：

- 快速判断
- 轻量简报
- 深度档案

## 语言

默认使用中文沟通，并用中文编写调研文档。代码、命令、配置键、项目名、专有名词和必要引用可以保留原文。

## 研究模式

### `quick-answer`

默认模式。适合回答：

- 项目是做什么的。
- 解决什么问题。
- 是否值得关注或继续研究。
- 最小安装/使用入口。
- 和同类方案的表层差异。

规则：

- 不克隆源码。
- 不写文件。
- 不更新项目索引。
- 不提交 Git commit。
- 优先基于 GitHub 页面、README、官方文档和必要网络搜索回答。
- 回答中标注信息来源，并说明未读源码或未本地运行的限制。

### `brief`

轻量沉淀模式。只有当用户明确说“记录下来”“生成简报”“做个初步调研”“沉淀一份文档”等类似需求时使用。

规则：

- 默认不克隆源码。
- 生成或更新一份 `docs/projects/<project-id>/brief.md`。
- 更新 `docs/projects/index.md`，标注为“简报”。
- 完成后自动检查改动，并只提交本次 brief 相关文件。
- 不生成固定四文档。

`brief.md` 只回答“是否值得继续研究或采用”的轻量问题，包含：

- 高信号结论。
- 项目定位。
- 解决的问题。
- 典型使用场景。
- 安装/使用入口，只写最小可用路径。
- 亮点与风险。
- 适合继续深挖的问题。
- 信息来源与未验证项。

不要把 `brief.md` 写成四份深度文档的压缩版；不要包含完整源码结构、完整安装排障、完整 API/命令清单或大段 README 翻译。

### `deep-dossier`

深度档案模式。只有当用户明确要求“深入研究”“源码结构”“架构分析”“调用链”“完整档案”“系统调研”或类似目标时使用。

规则：

- 可以浅克隆源码、阅读关键文件、阅读测试和 examples。
- 可以按目标验证安装、构建或基础运行，但必须控制成本和风险。
- 按研究目标选择文档，不固定生成四份。
- 更新 `docs/projects/index.md`。
- 完成后自动检查改动，并只提交本次 deep-dossier 相关文件。

文档选择：

- 默认深度调研：生成或更新 `overview.md`。
- 用户关注安装、配置、部署或使用：生成或更新 `usage.md`。
- 用户关注源码结构、调用链、模块边界或设计决策：生成或更新 `architecture.md`。
- 用户明确要求完整档案或系统调研：可以生成 `overview.md`、`usage.md`、`architecture.md`、`notes.md`。
- `notes.md` 只在确实有未解问题、风险、后续研究方向或阅读笔记需要沉淀时生成。

## 意图分流

看到仓库地址后，先判断用户意图：

- “看看”“研究”“分析”“总结”“值不值得用”“怎么用” -> `quick-answer`
- “记录”“生成文档”“沉淀”“简报”“初步调研” -> `brief`
- “深入”“源码”“架构”“调用链”“完整档案”“系统调研” -> `deep-dossier`

如果存在多种解释，先说明可能路径和推荐路径，再询问用户确认。不要默默选择更重的模式。

## 升级规则

允许从轻量模式升级，但必须说明原因并征求确认：

- 从 `quick-answer` 升到 `brief`：说明需要写入文档并询问确认。
- 从 `quick-answer` 或 `brief` 升到 `deep-dossier`：说明 README、官方文档或公开资料不足以回答哪个关键问题，需要浅克隆源码确认什么，再询问“是否确认执行？”。
- 通过 GitHub 页面、官方文档或网络搜索补充信息不算升级，不需要额外确认，但要标注来源和不确定性。

不要因为“研究”两个字就自动克隆源码。

## 信息源优先级

`quick-answer` 和 `brief` 默认按这个顺序取材：

1. GitHub 仓库主页元数据：简介、stars、license、最近提交、release 等。
2. README、官方文档、docs 站点。
3. package、build、config 文件的网页内容，例如 `package.json`、`pyproject.toml`、`Cargo.toml`，只在需要确认技术栈或安装方式时读取。
4. 网络搜索：用于补充项目背景、同类比较或公开反馈；必须标注来源。
5. 源码仓库浅克隆：不在 `quick-answer` 自动发生，只在升级到 `deep-dossier` 并得到确认后使用。

仓库元数据会随时间变化；当 stars、license、release、最新提交、维护状态等影响结论时，需要浏览或用其他方式确认最新状态。

## 工作流程

### 1. 确认仓库身份

识别并记录：

- 来源平台和仓库地址。
- owner 与 repo 名。
- 默认分支。
- 许可证。
- stars、forks、releases、最新提交等可能变化的公开元数据。

### 2. 创建稳定项目 ID

文档目录使用：

```text
docs/projects/<host>-<owner>-<repo>/
```

规范化规则：

- 全部转为小写。
- 非字母数字字符折叠为短横线。
- 去掉首尾短横线。

示例：

- `https://github.com/mattpocock/skills` -> `docs/projects/github-mattpocock-skills/`
- `https://github.com/zilliztech/claude-context` -> `docs/projects/github-zilliztech-claude-context/`

不要使用裸仓库名作为目录名，例如 `skills`、`docs`、`app`、`server`、`cli`、`core`，除非用户明确要求。

### 3. 高效阅读项目

优先阅读：

- `README`
- 官方文档
- examples
- release notes / changelog
- package、build、config 文件
- license
- contribution 文档
- 入口文件和核心模块
- 能揭示预期行为的测试

只有在 `deep-dossier` 模式才默认进行本地源码导航。本地源码导航优先使用 `rg` 和 `rg --files`。不要无目的通读所有源码；先建立入口、核心模块、适配层、测试、示例和构建脚本索引。

### 4. 合理验证使用方式

默认只做低风险静态研究。安装依赖、构建、运行测试或启动服务，必须和用户目标直接相关，并且成本可控。

可以直接做：

- 浅克隆源码。
- 读文件。
- `rg`、`rg --files`。
- 查看 package/config。
- 阅读测试和 examples。

谨慎做：

- 安装依赖。
- 构建。
- 运行测试。
- 启动服务。

不做或再次确认：

- 需要凭据。
- 下载大模型或大数据。
- 启动外部服务。
- 写入系统目录。
- 调用付费 API。
- 长时间构建或运行。

如果执行验证，记录：

- 尝试过的命令。
- 必要运行时和依赖。
- 必要环境变量和配置。
- 结果或阻塞原因。

不要把整个会话耗在安装排障上。如果因为凭据、网络、重依赖、本地服务、平台限制或潜在破坏性副作用无法运行，应停止并记录阻塞。

### 5. 区分事实和判断

重要结论尽量绑定来源：

- 文件路径
- 命令结果
- 官方链接
- 公开页面

清楚标注：

- `事实`
- `推断`
- `个人判断`
- `尚未验证`

不要把 README 的宣传语直接当作已验证能力。

## 文档产出

### 项目索引

所有写入 `brief` 或 `deep-dossier` 的项目，都维护在：

```text
docs/projects/index.md
```

索引使用表格，推荐字段：

- 项目
- 类型
- 一句话定位
- 主要文档
- 最近更新

根目录 `README.md` 只保留索引入口，不维护具体项目清单。

### `overview.md`

包含：

- 高信号总结。
- 项目定位。
- 目标用户。
- 核心能力。
- 适用场景。
- 不适用边界。
- 技术栈。
- 初步评价。
- 仓库来源、默认分支、许可证和调研快照。

适用于 `deep-dossier`。不要在 `brief` 模式强制创建。

### `usage.md`

包含：

- 安装和设置。
- 配置方式。
- 常用命令。
- 运行环境要求。
- 已尝试的验证。
- 未验证内容及原因。
- 常见失败原因。

只有用户关注安装、配置、运行或部署时创建或更新。

### `architecture.md`

包含：

- 源码结构概览。
- 入口文件。
- 核心模块。
- 数据流或控制流。
- 从代码/文档可见的设计决策。
- 构建、测试、发布流程。
- 架构风险或取舍。

流程复杂时可以使用 Mermaid 图。

只有用户关注源码结构、调用链、模块边界或设计决策时创建或更新。

### `notes.md`

包含：

- 已阅读材料。
- 调研快照。
- 关键发现。
- 未解问题。
- 风险和限制。
- 后续研究方向。
- 个人评价。

只在确实需要沉淀阅读笔记、未解问题、风险或后续研究方向时创建或更新。

## 自动 commit

自动 commit 只适用于实际写入研究产物的任务：

- `quick-answer`：不写文件，不提交。
- `brief`：写入 `brief.md` 或更新 `docs/projects/index.md` 后，自动检查改动并提交本次 brief 相关文件。
- `deep-dossier`：写入或更新研究文档后，自动检查改动并提交本次 deep-dossier 相关文件。

提交前必须确认只包含本次任务相关文件，不要把用户已有的无关修改一起提交。

修改 skill、README、AGENTS 或项目工作流本身，不因为本规则自动 commit；只有用户明确要求提交时才提交。

## 最终回复

最终回复包含：

- 一段高信号结论。
- 当前模式和为什么选择该模式。
- 已创建或更新的文件链接；`quick-answer` 模式说明未写文件。
- 未完成的验证。
- 重要不确定性。
- 如果创建了 commit，说明 commit 信息。

保持简洁；详细调研内容放在文档文件里。
